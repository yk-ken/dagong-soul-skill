# 阶段感知与策略路由规则

本文件定义用户的阶段判定规则和各阶段的策略路由逻辑。phase_router 是策略分析的第一个环节，决定后续分析的侧重点。

---

## 一、阶段定义

### 阶段总览

| 阶段码 | 显示名 | 含义 |
|--------|--------|------|
| `negotiating` | 协商中 | 公司已与用户约谈，双方正在沟通中 |
| `pressured` | 被施压 | 公司加大施压力度，可能涉及威胁、强制要求等 |
| `noticed` | 已通知 | 公司已出具书面解雇通知 |
| `separated` | 已离职 | 用户已实际离开公司 |
| `arbitrating` | 仲裁中 | 已正式提交劳动仲裁申请 |
| `closed` | 已结案 | 仲裁裁决已下达或双方达成和解 |

---

### 阶段详细定义

#### 1. `negotiating` —— 协商中

**进入条件（满足任一）：**
- 首次被 HR 或直属领导约谈
- 公司口头表达希望用户离职
- 公司提出协商解除劳动合同的意向
- 用户主动提出希望了解离职补偿

**退出条件（满足任一）：**
- 收到公司出具的书面解除劳动合同通知 → 进入 `noticed`
- 双方达成协商解除协议 → 进入 `closed`
- 沟通完全破裂，公司明确不再协商 → 根据情况进入 `noticed` 或 `pressured`
- 用户主动离职 → 进入 `separated`

**阶段信号词（用户对话中出现的标志性表述）：**
- "HR找我谈话了"
- "领导说我不合适"
- "公司想让我走"
- "约谈了"
- "说试用期不过"

---

#### 2. `pressured` —— 被施压

**进入条件（在 `negotiating` 基础上，满足任一）：**
- 公司多次约谈，频率明显增加
- 公司施加心理压力（"不签以后更麻烦""对你以后找工作不好"）
- 公司威胁或暗示不利后果
- 公司要求限期签字（如"今天不签就没这个方案了"）
- 公司安排交接工作或收回权限
- 公司以停岗、降薪等方式施压

**退出条件（满足任一）：**
- 收到书面解除通知 → 进入 `noticed`
- 用户签字离职 → 进入 `separated`
- 用户自行离职 → 进入 `separated`
- 施压停止，回到正常协商 → 回到 `negotiating`

**阶段信号词：**
- "说不签对我不好"
- "让我今天就签"
- "威胁我"
- "停了我的系统权限"
- "不让上班了"
- "说会通知新公司"
- "逼迫我签字"

---

#### 3. `noticed` —— 已通知

**进入条件（满足任一）：**
- 收到公司出具的书面《解除劳动合同通知书》
- 收到公司邮件或正式函件通知解除
- 收到公司通过微信/钉钉发送的正式解除通知（需能明确体现公司单方解除的意思表示）

**退出条件（满足任一）：**
- 用户实际离职 → 进入 `separated`
- 公司撤回解除通知 → 回到 `negotiating`
- 用户申请仲裁 → 进入 `arbitrating`

**阶段信号词：**
- "收到了解除通知"
- "公司发了解雇通知书"
- "书面通知我了"
- "邮件通知解除"
- "给了我一张纸"

---

#### 4. `separated` —— 已离职

**进入条件（满足任一）：**
- 已办理离职交接手续
- 已被要求离开工作场所
- 公司已停止发放工资
- 已被移出工作群组/系统
- 已拿回个人物品

**退出条件（满足任一）：**
- 提交劳动仲裁申请 → 进入 `arbitrating`
- 超过仲裁时效（1年） → 进入 `closed`（时效已过，权利丧失）

**阶段信号词：**
- "已经离职了"
- "走人了"
- "不在那干了"
- "已经离开公司"
- "交接完了"
- "被赶出来了"
- "工资停了"

---

#### 5. `arbitrating` —— 仲裁中

**进入条件：**
- 已向劳动争议仲裁委员会提交仲裁申请书并获得受理

**退出条件（满足任一）：**
- 仲裁裁决下达 → 进入 `closed`
- 仲裁调解成功 → 进入 `closed`
- 撤回仲裁申请 → 回到 `separated`

**阶段信号词：**
- "已经申请仲裁了"
- "仲裁委受理了"
- "等开庭"
- "仲裁进行中"

---

#### 6. `closed` —— 已结案

**进入条件（满足任一）：**
- 仲裁裁决已下达
- 仲裁调解成功并签署调解书
- 双方达成和解协议并履行完毕
- 法院判决生效
- 超过仲裁时效（1年）且未申请仲裁

**退出条件：** 无（终态）

**阶段信号词：**
- "仲裁结果下来了"
- "调解成功了"
- "判了"
- "已经拿到钱了"
- "和解了"

---

## 二、各阶段策略重心

### `negotiating` 协商中

**核心策略：谈判话术 + 底线设定 + 暗中收集证据**

| 维度 | 策略要点 |
|------|---------|
| 谈判话术 | 准备应对公司各类话术的回应；保持冷静、专业；不暴露底线 |
| 底线设定 | 根据法律规定计算合理范围（N 或 2N）；设定理想值和最低可接受值 |
| 证据收集 | **此阶段是收集证据的黄金期**：录音谈话、保存聊天记录、截图考勤、备份工作邮件 |
| 心态建议 | 不要急于表态；每次约谈后记录要点；不要被"今天就签"的压力催促 |
| 时间管理 | 不必急于答应，利用时间差收集更多证据和思考策略 |

**此阶段关键输出：**
- 底线金额计算
- 谈判话术清单
- 证据收集行动清单
- 红线提醒（绝对不签自愿离职）

---

### `pressured` 被施压

**核心策略：红线提醒 + 录音建议 + 记录模板**

| 维度 | 策略要点 |
|------|---------|
| 红线提醒 | 反复强调不可逾越的红线：不签自愿离职、不口头同意、不主动交 resignation |
| 录音建议 | 所有与 HR/领导的谈话必须全程录音；提供录音操作指引 |
| 记录模板 | 提供谈话记录模板：时间、地点、在场人员、公司表述、我的回应 |
| 法律武器 | 明确告知公司施压行为的法律后果；引用法条增强底气 |
| 安全保障 | 如涉及人身威胁，建议报警并保存记录 |

**此阶段关键输出：**
- 红线清单（紧急版）
- 录音操作指引
- 谈话记录模板
- 应对施压的话术
- 公司施压行为的法律定性

---

### `noticed` 已通知

**核心策略：书面回复模板 + 证据保全**

| 维度 | 策略要点 |
|------|---------|
| 书面回复 | 3日内回复公司，明确表示不同意解除决定；提供回复模板 |
| 证据保全 | 立即保全所有工作证据；备份电子文件；截图系统数据 |
| 通知审查 | 审查解除通知的内容：理由是否合法、程序是否合规 |
| 权利告知 | 告知用户公司需承担的举证责任 |
| 时效启动 | 解除通知上的日期是仲裁时效起算点，开始倒计时 |

**此阶段关键输出：**
- 书面回复函模板
- 解除通知合法性审查报告
- 紧急证据保全清单
- 仲裁时效起算日期标注

---

### `separated` 已离职

**核心策略：仲裁申请书 + 证据清单 + 时效提醒**

| 维度 | 策略要点 |
|------|---------|
| 仲裁准备 | 协助撰写仲裁申请书；明确仲裁请求和事实理由 |
| 证据整理 | 按仲裁要求整理证据清单；编号、分类、写明证明目的 |
| 时效提醒 | 以醒目方式标注仲裁时效截止日期 |
| 金额核算 | 精确计算可主张的各项金额，附计算过程 |
| 材料清单 | 列出提交仲裁所需的全部材料清单 |

**此阶段关键输出：**
- 仲裁申请书草稿
- 证据清单（编号版）
- 金额计算明细
- 仲裁材料清单
- 时效倒计时提醒

---

### `arbitrating` 仲裁中

**核心策略：陈述词 + 答辩策略**

| 维度 | 策略要点 |
|------|---------|
| 陈述词 | 协助撰写仲裁庭陈述词：事实陈述、法律依据、请求事项 |
| 答辩策略 | 预测公司可能的答辩理由，准备反驳要点 |
| 举证质证 | 准备举证提纲；练习质证话术 |
| 开庭准备 | 开庭注意事项、着装建议、流程说明 |
| 调解应对 | 仲裁庭调解阶段的应对策略；调解底线 |

**此阶段关键输出：**
- 仲裁陈述词草稿
- 答辩反驳要点
- 举证质证提纲
- 开庭准备清单
- 调解底线金额

---

### `closed` 已结案

**核心策略：强制执行 + 经验沉淀**

| 维度 | 策略要点 |
|------|---------|
| 裁决解读 | 解读裁决书内容，明确各方权利义务 |
| 强制执行 | 如公司不履行裁决，协助准备强制执行申请 |
| 经验沉淀 | 总结维权经验，可分享给其他有类似遭遇的劳动者 |
| 情绪关怀 | 肯定用户的勇气和付出；提供心理调适建议 |

**此阶段关键输出：**
- 裁决解读说明
- 强制执行申请指引（如需要）
- 维权经验总结模板

## 三、用户意愿优先决策树

### 决策流程

```
用户进入系统
 │
 ├─ Step 1: 判定当前阶段
 │   └─ 根据 phase_router 规则判定阶段
 │
 ├─ Step 2: 给出全局推荐
 │   ├─ 基于阶段 + 证据充分度 + 胜诉概率
 │   ├─ 给出理性推荐方向
 │   └─ 说明推荐理由
 │
 ├─ Step 3: 询问用户意愿
 │   └─ "基于当前情况，我建议 {推荐方向}。您希望怎么做？"
 │       1. 协商解决（争取合理补偿）
 │       2. 坚决不离职（反驳公司决定）
 │       3. 申请仲裁（走法律程序）
 │       4. 还没想好（帮我分析各选项）
 │
 ├─ Step 4: 按选择执行
 │   ├─ 协商 → 分支 A（协商谈判策略）
 │   ├─ 不离职 → 分支 B（反驳策略）
 │   ├─ 仲裁 → 分支 C（仲裁准备）
 │   └─ 没想好 → 三分支对比 + cost_calculator 分析
 │
 └─ Step 5: 持续跟踪
     ├─ 用户可随时切换意愿
     ├─ 阶段变化时自动重新路由
     └─ 新证据/新事件时触发策略更新
```

### 阶段与意愿的推荐组合

| 阶段 | 默认推荐 | 推荐理由 | 替代方案 |
|------|---------|---------|---------|
| `negotiating` | 协商（分支 A） | 协商成本最低，证据收集期 | 如证据充分可直接准备仲裁 |
| `pressured` | 协商 + 准备仲裁（A→C） | 先尝试协商，同时做好仲裁准备 | 如公司明显违法可直奔仲裁 |
| `noticed` | 协商或仲裁（A/C） | 取决于胜诉概率和用户意愿 | 证据不足时优先补充证据 |
| `separated` | 仲裁（分支 C） | 已离职协商空间小，仲裁是主要路径 | 金额较小可考虑协商 |
| `arbitrating` | 继续仲裁 | 已进入程序 | 庭前调解如条件好可接受 |
| `closed` | 执行或总结 | 视结果而定 | 不服裁决可起诉 |

### 意愿切换规则

1. **用户可在任何时候切换意愿**，无需理由
2. 切换时：
   - 保留之前的分析结果（标记为"历史方案"）
   - 以新意愿重新生成策略
   - 提示用户新方案的差异点
3. **自动切换触发条件：**
   - 阶段变化（如从 `negotiating` 进入 `noticed`）
   - 关键证据变化（如获得关键录音）
   - 时效临近（如仲裁时效不足 3 个月）

## 四、阶段转换日志格式

每次阶段变化需记录：

```json
{
  "case_id": "string",
  "transition": {
    "from_phase": "string",
    "to_phase": "string",
    "timestamp": "ISO 8601",
    "trigger": "user_reported|system_detected|event_based",
    "trigger_detail": "string"
  },
  "strategy_adjustment": "string"
}
```

该日志追加到 `cases/{case_id}/phase_log.json` 中。
